# RFWG 收尾移交与多轮深化调研

> 供「收尾移交」(SKILL.md 第 9 步)与「在既有调研包上继续深化」时查阅。
> 来源:一次真实全流程的复盘——一期群聊调研(RFWG 标准流程)之后,同一个包又经历了 **8 轮深化调研、35 份过程报告、一份收敛稿主文档改到 v4.3、两次分享前对齐检查**;下面每条约定都对应一个实际踩过的坑。

## 一、收尾三件套(一期完成时就该产出)

### 1. `00-数据完整性核验.md`(数据边界清单,给后续深度调研会话看)

回答一个问题:**抓到的原始素材到底完整到什么程度**。后续任何深化会话、任何拿到包的外部 AI,都以它判断"什么可信、什么缺、要不要补采"。结构:

- 头部一句话总评(如"文字完整;图片仅 ~16% 有像素;朋友圈只有 X 一人完整");
- 总览表:每个数据维度一行(是否完整 ✅/⚠️/❌ + 一句说明);
- 分节详述:每个维度的条数/覆盖区间/截断检查/时间戳精度/已知坑(如"`--media` 解析图片路径有 bug,不能可靠映射");
- 「给深度调研会话的建议」:哪些可直接当全量信任、缺口怎么补(命令级)、引用格式约定(群聊分钟级 `[MM-DD HH:MM]`、朋友圈秒级)。

### 2. `README-导读.md`(分享包入口,人和 AI 共用)

- **怎么读(建议顺序)**:核验 → HTML 导览 → 细节 MD → 溯源目录;
- **一句话结论** + 关键提醒;**目录结构树**(每行一句用途);**关键数字表**(消息数/图片数/轮次数——数字只在此表维护,别处引用);
- **给 AI 的入口建议**:把包喂给 AI 时从哪个文件进、引用怎么核对、哪些是过程稿不要当结论引用;
- **转发前留意**:匿名化边界、哪些目录可整体删除不影响阅读(`原始数据/` 等)、含隐私文件清单。

### 3. HTML 报告的「后续产出」扩展位

调研包大概率会长大(见下)。一期 HTML 收尾时如果已知会有二期,预留或直接加一节「后续深度产出」(nav + section,列出收敛稿与过程报告索引);后补也行,但要记得**三个入口(README/HTML/00-核验)必须同步指向最新**——这是分享前检查的固定项。

## 二、多轮深化调研的组织约定(包会长大)

一期报告常引出深化需求(业界对标、源码考察、论文精读、红队、推理推演、对抗评审、落地评估……)。没有约定时的实际后果:计数漂移(语料 19/27/41 增补到 21/31/47 后一半文档没跟上)、失效指针(过程稿指向已重构掉的章节号)、被推翻的旧结论被重新引用。约定如下:

### 目录与编号

```
$OUT/
├── <收敛稿主文档>.md        # 唯一权威结论(活文档)
├── 调研补充/NN-主题.md      # 过程报告,顺序编号只增不改
├── 外部资料/                # 外部语料:开源项目/论文/文章 + README-资料清单.md
└── (README-导读 / 00-核验 / HTML 同步维护)
```

- 过程报告 `NN-` 顺序编号,正文用 `` `NN` `` 互相引用;
- `外部资料/README-资料清单.md`:每项一句"为什么相关"+三类计数;**增补语料时当场更新计数与清单行**,别欠账。

### 活文档 vs 过程文档(核心二分)

| 类型 | 哪些 | 规则 |
|------|------|------|
| **活文档** | 收敛稿主文档、README-导读、00-核验、HTML、外部资料清单 | 永远就地更新到最新;收敛稿头部维护一行式版本沿革(v4.1=什么、v4.2=什么…) |
| **过程文档** | `调研补充/NN-*` 全部 | **正文冻结不改**;状态变化只加**头部通告** |

头部通告模板(放标题下第一行,打开即知状态):

```
> ⚠️ **修订通告(日期)**:本篇 X 结论已被 `NN` 修正/推翻:…;权威表述见主文档(现 vX.Y)。
> 📌 **状态(日期)**:本篇发现已裁决并入 `NN`(现行 vX);过程稿保留原状。
```

注意:通告里的指针也会失效(主文档重构后章节号变了)——分享前检查要 grep 一遍所有通告的指针。

### 收敛稿主文档的三个装置

1. **文档地图**(表格):每轮一行——文件 / 轮次 / 状态(结论是否已回灌、头部有无通告、涉隐私提示);
2. **防复活清单**(附录):被推翻的旧表述 | 修正后表述 | 裁决出处——防止后续会话把中间结论当最终结论重新引用;
3. **版本沿革**:头部一行式 changelog,过程文档通告统一指"现 vX.Y"。

### 对抗评审工作法(可选,但对结论质量提升最大)

- **同模型 subagent 互审**:每轮产出派对抗评审逐条判决(成立/需限定/需修改/推翻),修正就地生效 + 原稿加通告;
- **多方互审**(评估外部方案时):正方 steelman / 反方证伪 / 中立对账 / 独立评审,四方交叉;
- **减法批判**(扩充整合之后必做):专门猎杀过度设计,逐条"砍/简化/保留"——扩充轮的通病是每路输入都想证明自己有产出,整合稿照单全收;
- **两条引用纪律**(高频翻车点):① **设计 ≠ 实施**——把 SoT/设计文档里的机制当"已实现"引用是最常见错误,引用时标注"设计,未实现/已实现于 X";② **AI 转述必须回原文核对**——实测外部 AI 读包后的错引全部朝利己方向偏,自己的转述同样要抽查回原文。

## 三、分享前对齐检查清单(每次对外分享前跑一遍)

1. **计数 sweep**:文档数/语料数/轮次数在 README、HTML、00-核验、收敛稿头部、资料清单五处一致(grep 旧数字确认清零);
2. **版本 sweep**:收敛稿版本号在所有入口与通告中一致;
3. **指针抽验**:`NN` 指针、章节号指针、文件路径逐一存在(`ls` 验证,重构后最易断);
4. **入口同步**:README 怎么读/HTML 后续节/00-核验注记三处指向最新产出;
5. **隐私再盘点**:**新增文件会引入新隐私面**(深化轮常引用自有项目内部细节/真实身份)——逐文件过一遍,在 README「转发前留意」维护"外发前可整体删除"清单,并确认收敛稿正文不依赖这些文件;
6. **通告覆盖巡检**:每份过程文档头部有状态说明(脚本抽 2-4 行 grep 通告关键词);
7. **竞态检查**:`ls -lat` 看有无本会话不知道的新文件/被改文件。

## 四、多会话与中断恢复纪律

- **动笔前先盘点目录**:以目录状态为准,不以会话记忆为准——被"打断"的 subagent 可能已在后台完成并落盘(实测:界面显示中断,4 份报告照常写完);
- **同一调研目录同一时间只允许一个会话收尾**:发现并行写入(mtime 突变、陌生编号文件)先停下盘点、消化对方产出,再继续;两个会话同时整合/裁剪同一文件必然互相覆盖;
- 大包分目录整理(`报告/`、`人物发言原文/`、`周度时间线/`、`原始数据/`)时,README 目录树同步更新;`周度时间线/`(按周切的深度时间线)对长跨度调研值得单独产出。
